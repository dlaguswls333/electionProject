# 웹개발프로그래밍 팀 프로젝트 개발 일지

## 프로젝트 개요

- 프로젝트명: 대소고 교내 스마트 선거 시스템
- 개발 형태: React + Vite 기반 웹 애플리케이션
- 주요 기능: 로그인/회원가입, 서버 후보 조회, 후보자 등록, 투표, 투표 완료 상태 저장, 투표 현황 집계, CSV 다운로드
- 작성 목적: 결과 보고서에 첨부할 개발 과정 기록

## 내 역할

이번 프로젝트에서 나는 프론트엔드 화면 구성과 인증 흐름 정리, 서버 API 연동을 중심으로 개발을 맡았다. 특히 로그인/회원가입 화면, 라우팅 구조, 후보자 등록 및 투표 화면 연결, 후보 조회 API와 투표 API 연동, 투표 완료 상태 유지 기능을 담당했다.

구체적으로 구현한 기능은 다음과 같다.

- React Router 기반 페이지 라우팅 구성
- 로그인 화면과 회원가입 화면 구현
- 학생 로그인 API 연동 구조 작성
- 회원가입 API 연동 구조 작성
- DAuth 인증 방식 제거 및 일반 로그인 방식으로 변경
- 관리자/학생 역할 기반 화면 접근 제어 정리
- 후보자 등록, 투표, 집계 화면을 라우트에 연결
- GitHub Pages 배포를 위한 Vite base 경로 및 SPA fallback 설정
- 로컬 개발 중 CORS 문제를 줄이기 위한 Vite API proxy 설정
- 서버 후보자 조회 API 연동
- 투표 제출 API 연동
- 새로고침 후 중복 투표를 막기 위한 사용자별 투표 완료 상태 저장
- 투표 완료 모달 구현

## AI 활용 및 직접 수정한 내용

개발 과정에서 AI를 활용해 React 컴포넌트 구조, 라우팅 분리 방식, API 연동 코드 초안을 빠르게 만들었다. 하지만 생성된 코드를 그대로 사용하지 않고 프로젝트 상황에 맞게 수정했다.

직접 수정한 대표적인 부분은 다음과 같다.

1. DAuth 제거
   - 처음에는 DAuth OAuth 방식으로 로그인 흐름을 만들었지만, 프로젝트 요구사항이 바뀌어 일반 로그인 방식으로 되돌렸다.
   - `DauthCallbackScreen.jsx`와 `src/auth/dauth.js`를 제거하고, `LoginScreen.jsx`를 일반 폼 로그인 방식으로 수정했다.

2. 라우팅 구조 개선
   - 처음에는 `App.jsx` 안에 모든 `<Route>`가 들어 있어 코드 가독성이 떨어졌다.
   - 이를 직접 확인한 뒤 `src/routes/AppRoutes.jsx`로 라우트 선언을 분리했다.
   - `App.jsx`는 상태와 이벤트 핸들러를 관리하고, `AppRoutes.jsx`는 화면 연결만 담당하도록 역할을 나누었다.

3. React Router 사용 방식 정리
   - 처음에는 모든 이동을 `useNavigate`로 처리하려 했지만, 라우트 접근 제어에는 `<Navigate>`를 쓰는 방식이 더 적절하다고 판단했다.
   - 로그인 성공, 로그아웃처럼 이벤트로 발생하는 이동은 `useNavigate`를 사용하고, 비로그인 접근 차단 같은 선언적 리다이렉트는 `<Navigate>`로 유지했다.

4. Swagger에 맞춘 API 요청 수정
   - Swagger 문서를 보고 로그인 요청 body가 `username`, `password`라는 것을 확인했다.
   - 기존 폼의 `account` 값을 서버 요청 시 `username`으로 변환하도록 수정했다.
   - 회원가입 요청도 Swagger에 맞춰 `username`, `password`, `name`만 전송하도록 구현했다.

5. CORS 문제 대응
   - ngrok 서버로 직접 요청할 때 브라우저 CORS 에러가 발생했다.
   - 서버 CORS 설정이 근본 원인이라는 점을 확인하고, 로컬 개발에서는 Vite proxy를 통해 `/api` 요청을 우회하도록 설정했다.
   - 실제 배포 환경에서는 백엔드에서 `Access-Control-Allow-Origin`을 설정해야 한다는 점을 정리했다.

6. 후보자/투표 API 연동
   - 기존에는 후보자와 투표 상태가 로컬 데이터 중심으로 동작했다.
   - Swagger에서 확인한 `/api/candidates`, `/api/votes` 명세에 맞춰 `src/api/election.js`를 만들고 API 요청을 한 파일로 모았다.
   - 서버 후보 응답에는 화면에서 쓰는 `number`, `color`, `description` 같은 필드가 없어서, 응답을 그대로 쓰지 않고 화면용 데이터로 변환하는 과정을 추가했다.

7. 중복 투표 방지와 완료 모달
   - 투표 후 새로고침하면 `hasVoted` 상태가 초기화되어 다시 투표할 수 있는 문제가 있었다.
   - 백엔드에 투표 여부 조회 API가 없는 상태라 프론트에서는 로그인 사용자별로 투표 완료 상태를 `localStorage`에 저장하도록 수정했다.
   - 투표 성공 직후에는 완료 모달을 띄워 제출 결과를 명확히 보여주고, 제출 버튼은 비활성화되도록 처리했다.

## 구현하면서 어려웠던 점

- 인증 방식이 DAuth에서 일반 로그인 방식으로 바뀌면서 기존 코드를 제거하고 다시 연결해야 했다.
- 로그인 후 뒤로 가기 동작이 왜 로그인 화면으로 돌아가지 않는지 확인해야 했다.
- React Router에서 `useNavigate`와 `<Navigate>`를 언제 써야 하는지 구분하는 과정이 필요했다.
- ngrok 서버와 로컬 프론트엔드 주소가 달라 CORS 에러가 발생했고, 프론트와 백엔드의 책임 범위를 구분해야 했다.
- 투표 API가 `candidateId` 숫자를 요구하는데 로컬 후보 ID가 문자열이라 Bad Request가 발생할 수 있었다.
- 서버 후보 응답과 기존 프론트 화면 데이터 구조가 달라 그대로 연결하면 후보 번호, 색상, 설명 필드가 맞지 않았다.
- 새로고침하면 React 상태가 초기화되어 투표 완료 여부가 사라지는 문제가 있었다.

## 해결 방법

- 인증 관련 코드를 `src/api/auth.js`로 분리해 로그인/회원가입 요청을 한 곳에서 관리했다.
- 후보자 조회, 후보자 등록, 투표 제출 코드를 `src/api/election.js`로 분리해 선거 관련 API 요청을 한 곳에서 관리했다.
- 라우팅 코드를 `AppRoutes.jsx`로 분리해 `App.jsx`의 가독성을 개선했다.
- 로그인 상태는 `sessionStorage` 또는 `localStorage`에 저장하고, 앱 시작 시 복원되도록 했다.
- 투표 완료 상태도 사용자별 `localStorage` 키로 저장해 새로고침 후에도 재투표를 막았다.
- 로컬 개발에서는 Vite proxy를 사용해 CORS 문제를 우회했다.
- `npm run lint`, `npm run build`를 실행해 문법 오류와 빌드 오류가 없는지 확인했다.

## KPT 프로젝트 리뷰

### Keep

- 컴포넌트를 화면 단위와 재사용 단위로 나누어 관리한 점은 유지하고 싶다.
- 라우팅과 API 요청을 별도 파일로 분리한 뒤 코드 읽기가 쉬워졌다.
- Swagger를 확인하면서 실제 서버 요청 형식에 맞춰 프론트 코드를 수정한 점이 좋았다.
- 투표 완료 상태처럼 새로고침 후에도 유지되어야 하는 값과 화면에서만 필요한 선택 상태를 구분한 점은 유지하고 싶다.

### Problem

- 처음 인증 방식을 확정하지 못해 DAuth 코드를 작성했다가 제거하는 일이 발생했다.
- 커밋 메시지 중 `commit 1`처럼 의미가 부족한 메시지가 있어 개발 과정을 한눈에 파악하기 어렵다.
- 서버 CORS 설정과 프론트 proxy 설정의 차이를 처음에는 명확히 구분하지 못했다.
- 백엔드에 투표 여부 조회 API가 없어서 중복 투표 방지를 프론트 저장소에 의존해야 했다.
- 서버 데이터 구조와 프론트 화면 데이터 구조가 달라 중간 변환 로직이 필요했다.

### Try

- 다음 프로젝트에서는 API 명세와 인증 방식을 먼저 확정한 뒤 화면 개발을 시작하고 싶다.
- 커밋 메시지를 기능 단위로 더 구체적으로 작성하겠다.
- CORS, 인증 토큰 저장, 라우팅 보호처럼 프론트와 백엔드가 함께 맞춰야 하는 부분은 초기에 팀원과 먼저 확인하겠다.
- 중복 투표 방지는 최종적으로 서버에서 사용자별 투표 여부를 검증하도록 백엔드 팀과 API를 맞추겠다.

## 발표 때 설명할 내용

발표에서는 내가 구현한 로그인/회원가입 흐름과 라우팅 구조를 중심으로 설명할 예정이다.

- `/login`에서 `username`, `password`를 입력하면 `/api/auth/login`으로 요청을 보낸다.
- 응답으로 받은 `accessToken`, 사용자 정보를 저장하고 투표 화면으로 이동한다.
- `/signup`에서는 `username`, `password`, `name`을 입력해 `/api/auth/signup`으로 요청한다.
- 로그인 화면과 회원가입 화면은 버튼으로 서로 이동할 수 있다.
- `AppRoutes.jsx`에서 전체 페이지 라우팅을 관리하고, `RouteGuards.jsx`에서 로그인 여부에 따라 접근을 제한한다.
- `/api/candidates`로 후보 목록을 불러와 투표 화면에 표시한다.
- `/api/votes`로 선택한 후보 ID를 전송해 투표를 제출한다.
- 투표 성공 후 사용자별 투표 완료 상태를 저장해 새로고침 후에도 재투표를 막고 완료 모달을 보여준다.
- AI가 만든 초안을 그대로 제출한 것이 아니라, DAuth 제거, Swagger 요청 형식 반영, 라우팅 분리, CORS proxy 설정, 후보/투표 API 연동, 투표 완료 상태 저장 로직을 직접 수정했다.

## 다른 조 발표 피드백

다른 조 발표를 볼 때는 단순히 화면이 예쁜지보다 실제 사용 흐름이 끊기지 않는지를 중심으로 확인할 예정이다. 특히 로그인 후 접근 제어가 자연스러운지, 서버 API 실패 상황에서 사용자에게 안내가 되는지, 새로고침이나 뒤로 가기 같은 기본 브라우저 동작에서도 상태가 안정적으로 유지되는지를 볼 것이다.

피드백을 남길 때는 다음 기준으로 정리하겠다.

- 좋았던 점: 화면 구성, 기능 완성도, 발표자가 자기 코드 흐름을 이해하고 설명한 부분
- 개선할 점: API 에러 처리, 상태 유지, 라우팅 흐름, 모바일 화면 대응처럼 실제 사용 중 문제가 될 수 있는 부분
- 우리 프로젝트에 반영할 점: 다른 조가 사용한 검증 방식이나 UI 흐름 중 우리 선거 시스템에도 적용할 만한 아이디어
